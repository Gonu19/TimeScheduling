### 1. 팀 생성 및 접속 (Session Management)

- **1.1 팀 생성 (`POST /sessions`)**: 관리자가 방 이름과 목적(회의/근무)을 설정하고 UUID를 발급받습니다. (작성하신 내용과 동일)
    
- **[추가] 1.2 세션 정보 조회 (`GET /sessions/{sessionId}`)**: 유저 접속 시 필요한 필수 API입니다. 로그인 과정이 없으므로, 익명 링크를 타고 들어온 팀원에게 프론트엔드가 "이 방은 OOO 팀의 회의 방입니다"라고 화면을 그려주려면 해당 UUID의 메타 정보(방 이름, 목적, 만료일 등)를 불러와야 합니다.
    

### 2. 가용시간 등록 (Availability Handling)

- **2.1 가용시간 등록 (`POST /sessions/{sessionId}/availabilities`)**: 팀원이 닉네임과 48자리 비트 패턴 문자열을 서버로 전송합니다. 백엔드는 이를 64비트 정수로 파싱하여 저장합니다.
    
- **[수정/확장] 2.2 가용시간 현황 및 이전 데이터 조회 (`GET /sessions/{sessionId}/availabilities`)**: '이전 가용시간 불러오기'를 달성하기 위한 API입니다. 프론트엔드가 특정 유저(alias)의 기존 입력 데이터를 불러오거나, 현재까지 누가 입력을 완료했는지 팀원 현황 목록을 화면에 렌더링하기 위해 필요합니다.
    

### 3. 스케줄 추천안 도출 및 확정 (Recommendation & Confirmation)

- **3.1 3가지 추천안 도출 (`GET /sessions/{sessionId}/recommendations`)**: 백엔드 인메모리에서 비트 논리곱(`&`) 연산을 통해 3가지 추천안(최대 참석률, 평준화, 연속성)을 반환합니다.
    
- **3.2 스케줄 최종 확정 (`POST /sessions/{sessionId}/confirm`)**: 관리자가 추천안을 확정합니다. 낙관적 락(버전)을 검증하여 동시성 오염을 방지합니다.
    

### 4. [추가] 결과 조회 (Dashboard)

- **4.1 최종 확정 스케줄 조회 (`GET /sessions/{sessionId}/result`)**: 3.2에서 확정이 끝난 후, 모든 팀원들이 링크에 다시 접속했을 때 "우리 팀의 최종 일정은 이렇습니다"라고 보여줄 대시보드용 조회 API가 비어있습니다. 세션의 최종 확정 여부와 확정된 스케줄 슬롯 정보를 반환하는 엔드포인트가 반드시 추가되어야 합니다.

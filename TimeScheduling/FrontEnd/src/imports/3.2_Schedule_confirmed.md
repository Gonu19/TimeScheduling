### [API 3.2] 스케줄 최종 확정 (POST `/api/sessions/{sessionId}/confirm`)

▼ **Header**

- `Content-Type: application/json` : mandatory
    
- `Authorization: Bearer <adminToken>` : **mandatory**
    

▼ **Request**

- Path Variable
    
    - `sessionId` (string, UUID 형식): 확정할 세션 고유 ID
        
- Request Body (object)
    
    - `recommendationType` (string): 방장이 최종 선택한 추천안의 타입 식별자
        
    - `version` (integer): [API 3.1]에서 응답받았던 데이터 버전 (Concurrency control 용도)
        

JSON

```
{
    "recommendationType": "MAX_ATTENDANCE",
    "version": 3
}
```

▼ **Response** 성공적인 확정 처리 결과 (object)

- `status` (string): 세션의 변경된 상태
    
- `message` (string): 클라이언트에 노출할 성공 메시지
    
- 
JSON

```
{
    "status": "CONFIRMED",
    "message": "스케줄이 최종 확정되었습니다."
}
```

▼ **Error**

- `errorCode` (string): 시스템 에러 코드
    
- `message` (string): 에러 상세 메시지
    

JSON

```
{
    "errorCode": "ERR_CONCURRENCY_CONFLICT",
    "message": "추천안을 검토하는 동안 다른 팀원이 가용시간을 수정했습니다. 최신 데이터를 기반으로 추천안을 다시 도출해주세요."
}
```

▼ **예외 (Exception Handling)**

1. **동시성 충돌 (Conflict):** 클라이언트가 보낸 `version`과 DB에 Persist(영속화)된 Session 엔티티의 `@Version` 값이 불일치할 경우 JPA `OptimisticLockException`을 캐치하여 **409 Conflict** 반환. (Data integrity 무결성 보장)
    
2. **상태 이상 (Bad Request):** 이미 `"CONFIRMED"` 상태인 방에 대해 중복 확정 요청을 보낼 경우 **400 Bad Request** 반환.

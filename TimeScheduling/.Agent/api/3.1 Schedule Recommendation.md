
### [API 3.2] 스케줄 최종 확정 (POST /api/sessions/{sessionId}/confirm)

▼ **Description**

- 프론트엔드에서 관리자가 시간(Time)을 미세 조정(Manual Override)하거나 그대로 확정한 최종 스케줄 데이터를 서버에 영속화(Persist)합니다.
    
- MEETING 도메인은 배열에 단 1개의 객체만, WORK 도메인은 여러 개의 객체가 담겨 전송됩니다. 백엔드는 도메인 구분 없이 넘어온 Data Payload를 그대로 신뢰하여 덮어씌웁니다.
    

▼ **Header**

- `Content-Type: application/json`
    

▼ **Request Payload (`ConfirmScheduleRequest`)**

- `confirmedBlocks` (array of objects): 최종적으로 확정(및 시간 조정)된 타임 블록들의 배열
    
    - `startTime` (string): 조정이 완료된 최종 시작 시간 (Format: yyyy-MM-dd'T'HH:mm:ss)
        
    - `endTime` (string): 조정이 완료된 최종 종료 시간 (Format: yyyy-MM-dd'T'HH:mm:ss)
        
    - `assignedWorkers` (array of objects): 해당 시간에 배정된 근무자 명단 (프론트엔드에서 추천받은 명단 그대로 유지)
        
        - `id` (string, UUID): 참가자 고유 식별자
            
        - `name` (string): 참가자 이름
            
        - `isMandatory` (boolean): 필참 여부
            

**JSON Example (WORK 도메인에서 5/29 스케줄만 17:30으로 시간 조정을 한 경우)**

JSON

```
{
  "confirmedBlocks": [
    {
      "startTime": "2026-05-29T17:30:00", 
      "endTime": "2026-05-29T18:30:00",
      "assignedWorkers": [
        { "id": "uuid-1", "name": "고은우", "isMandatory": true }
      ]
    },
    {
      "startTime": "2026-05-30T17:00:00",
      "endTime": "2026-05-30T18:00:00",
      "assignedWorkers": [
        { "id": "uuid-2", "name": "김알바", "isMandatory": false }
      ]
    }
  ]
}
```

▼ **Response (200 OK)**

- 성공 시 별도의 Body 없이 HTTP Status 200만 반환하여 Network I/O를 최소화합니다.

### Architecture Strategy (도메인별 적용 전략)

- **회의 (MEETING):** 단일 블록(Single Block) 조율입니다. 대시보드에 띄워진 단 1개의 추천 일정(예: 14:00 ~ 16:00)의 시간을 앞뒤로 미세 조정하여 백엔드에 덮어씌웁니다.
    
- **근무 (WORK):** 다중 블록(Multiple Blocks) 조율입니다. 주간 근무표(Weekly Plan) 배열 속에서 **특정 요일의 특정 타임 슬롯 1개**만 콕 집어서(Targeting) 시간을 늘리거나 줄인 뒤, 전체 배열을 다시 백엔드로 통째로 밀어 넣습니다(Overwrite).
    

결국 백엔드는 프론트엔드가 건네주는 `confirmedBlocks` 리스트를 그대로 DB의 JSON 컬럼에 직렬화(Serialization)하여 저장만 하면 되는 구조가 됩니다.

### [API 3.3] Work Schedule Recommendation (GET /api/sessions/{sessionId}/work-recommendations)

▼ **Description**

- 오직 `domainType`이 "WORK"인 세션에서만 호출 가능한 Endpoint입니다.
    
- **Priority Assignment (우선 배정):** 알고리즘 로직 내부적으로 DB의 `isMandatory=true`인 멤버는 해당 타임 슬롯에 가용(Supply > 0)할 경우 최우선으로 할당(Allocated)됩니다.
    

▼ **Header**

- (GET 요청이므로 Body Content-Type은 불필요하지만, 토큰이 필요할 경우 Authorization 추가)
    

▼ **Request**

- **Path Variable**
    
    - `sessionId` (string, UUID 형식): 추천안을 조회하고자 하는 근무 일정 조율 방의 고유 ID
        

▼ **Response**

- Constraint Satisfaction Problem(제약 충족 문제) 알고리즘을 통해 도출된 주간 근무 계획안 배열 (array of objects)
    
    - `rank` (integer): 추천안 순위 (1~3위)
        
    - `totalCoverage` (string): 주간 전체 수요(Demand) 대비 실제 채워진 근무 시간의 충족률 백분율 (예: "100%")
        
    - `weeklyPlan` (array of objects): 해당 추천안에 포함된 다중 근무 타임 블록들
        
        - `startTime` (string): 근무 시작 시간 (Format: yyyy-MM-dd'T'HH:mm:ss, _KST 기준 파싱 필수_)
            
        - `endTime` (string): 근무 종료 시간 (Format: yyyy-MM-dd'T'HH:mm:ss, _KST 기준 파싱 필수_)
            
        - `assignedWorkers` (array of objects): 해당 타임 슬롯에 배정된 멤버 객체 명단 (동명이인 식별 및 UI 렌더링 용도)
            
            - `id` (string, UUID): 배정된 근무자의 고유 식별자
                
            - `name` (string): 근무자 이름
                
            - `isMandatory` (boolean): 매니저/필참 인력 여부 (프론트엔드에서 별표 ★ 뱃지 Read-only 표기용)

- `errorCode` (string): 시스템 에러 코드
    
- `message` (string): 에러 상세 메시지

▼ **예외 (Exception Handling)**

- **세션 부재 (Not Found):** 존재하지 않는 `sessionId` 요청 시 **404 Not Found** 반환.
    
- **도메인 불일치 (Bad Request / Conflict):** DB 상의 `domainType`이 "MEETING"인데 `[API 3.3]`을 호출한 경우 **400 Bad Request** 반환.
    
- **추천 불가 (Unprocessable Entity):** 관리자의 Demand(수요)를 100% 충족하는 근무표 조합이 단 하나도 도출되지 않을 경우(Hard Constraint 위반), 빈 배열(`[]`)을 반환하여 Client 측에서 "조건을 만족하는 근무표가 없습니다" 처리 유도.
    

### 💡 Backend Engineering Insight

- **Decoupling(디커플링)의 미학:** 이제 `[API 3.1]`과 `[API 3.3]`은 완전히 분리되어 각자의 도메인 로직에만 집중합니다. 코드가 간결해지며, 특정 도메인의 유지보수가 다른 도메인에 Side effect(부수 효과)를 일으키지 않습니다.
    
- **Immutable State 활용:** `assignedWorkers` 배열 내부에 `isMandatory` 속성을 포함시켜 응답(Response)함으로써, 프론트엔드는 복잡한 상태 관리(State Management) 없이 넘어온 배열만 순회하며 조건부 렌더링(Conditional rendering)을 깔끔하게 구현할 수 있습니다. 마치 견고한 배당주 포트폴리오(Dividend Portfolio)가 안정적인 캐시플로우(Cash flow)를 창출하듯, 탄탄한 Data payload가 프론트엔드의 렌더링 부담을 덜어줍니다.
